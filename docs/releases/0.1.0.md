# Tugboat 0.1.0 Release Notes

## Summary

Tugboat 0.1.0 is the first proposal-only MVP release. It ships the local CLI loop for indexing instruction files, ingesting trace bundles, running governed `llmff` audit/propose/eval pipelines, producing review artifacts, and keeping auto-apply disabled by default.

## User-Visible Changes

- Added `tugboat doctor`, `index`, `audit`, `propose`, `eval`, `report`, `optimize`, `status`, `harness check`, MCP contract tools, daemon queue commands, and operations commands.
- Added Markdown instruction discovery for `CODEX.md`, `AGENTS.md`, `SKILL.md`, runbooks, eval definitions, and policy docs.
- Added schema-validated artifacts for audit, candidate, eval, decision, report, optimization summary, operations, and release manifest outputs.
- Added review-gated apply and rollback workflows. These are not the default release posture.
- Added SkillOpt-style optimize memory flow: episode minibatch guidance is fed into `patch-propose`, and optimization-gate rejections persist bounded-edit fingerprints for later suppression.
- Added inline optimization validation metrics to generated reports so trigger score, held-out score, governance result, and recommendation are visible without opening the JSON artifact.
- Added first-class policy event groups to canonical episode artifacts so policy failures remain replayable evidence, not only generic event rows.
- Added review-only harness cleanup proposal artifacts, backed by structural eval evidence, so doc-gardening candidates enter a durable review queue without mutating instruction files.
- Added harness detection for broken repo-local file links in instruction maps and referenced runbooks, not only Markdown document links.
- Added MCP audit events for repo-bearing JSON-RPC validation failures so malformed tool calls still leave an auditable local trail.
- Added release-manifest secret scanning for retained evidence logs before release manifests are written.
- Added daemon queue state to operations observability summaries so release review can see outstanding daemon work.
- Added audit-ledger hardening so daemon queue jobs discovered from the sidecar queue are mirrored into the repo-local audit table, and candidate state changes point at the event that produced the current state.
- Added release-ops hardening so backup and restore plans reject `.sidecar`-internal archive/staging paths, and release manifests require retained pytest coverage evidence.
- Tightened daemon state transitions so audit/proposal jobs move directly to review, while eval jobs distinguish rejected candidates from daemon failures.
- Added checkpoint-carried daemon resume metadata support so interrupted `llmff` jobs can restart from validated checkpoint state without duplicating executable paths in the queue payload.

## Policy Changes

- Default posture is `proposal_only`.
- `auto_apply` remains disabled by default.
- Provider-backed `llmff` runs require explicit repository policy.
- Live provider smoke checks are opt-in and cannot be enabled by environment variables alone.
- Deterministic gates reject protected policy weakening, unsafe sections, stale base hashes, overlapping edits, malformed Markdown, and restricted instruction changes.

## Storage Changes

- Tugboat writes local state under `.sidecar`.
- SQLite audit events are append-only and hash-chained.
- Run artifacts are stored under `.sidecar/runs/<run-id>/`.
- Operations evidence is stored under `.sidecar/ops` and `.sidecar/ci`.

## Rollback Notes

Rollback means returning users to the prior package and preserving `.sidecar` readability:

```bash
python -m pip install tugboat==<previous-version>
tugboat status --repo .
```

If `.sidecar` data must be restored, use `docs/ops/sidecar-backup-restore.md`.
