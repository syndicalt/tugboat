# Tugboat v1.0.0 Roadmap

## Purpose

This roadmap turns the 0.1.0 production foundation into a stable v1.0 release that teams can adopt as a local sidecar for governed agent instruction maintenance.

Tugboat v1.0 should be the trusted local system for continuously observing, evaluating, and maintaining the operational files that shape coding agents: `CODEX.md`, `AGENTS.md`, `SKILL.md`, runbooks, policies, eval definitions, and related harness config.

The core posture does not change for v1.0:

- local-first operation;
- proposal-only by default;
- narrow Class A auto-apply only after explicit policy opt-in;
- evidence-gated proposals;
- human oversight as the primary control;
- VCS-backed mutation and rollback;
- `llmff` as the bounded pipeline runner, not Tugboat's authority layer.

## Current Baseline

The 0.1.0 foundation already includes the major product skeleton:

- CLI workflows for `doctor`, `init`, `status`, `index`, `audit`, `propose`, `eval`, `optimize`, `apply`, `auto-apply`, `rollback`, `mcp`, `daemon`, `harness`, `ci`, `retention`, and `ops`.
- Trace ingestion and canonical episode capture for generic JSONL, Codex, Claude, MCP session, and CI failure traces.
- Governed proposal loop with `llmff inspect`, file-backed `llmff run` artifacts, deterministic policy gates, eval reports, optimization summaries, and review reports.
- VCS-gated apply modes: proposal, branch, commit, and pull request.
- Rollback with recorded provenance, decision trace, apply plan, post-rollback hashes, and audit events.
- Narrow auto-apply lanes for `docs_hygiene` and `skill_improvement`, disabled by default and gated by policy, confirmation, ledger metrics, eval evidence, VCS, and rollback.
- Local daemon with queueing, run-once/cycle operations, read-only kill switch, stale lease handling, and local-only socket constraints.
- Read-first MCP surface for status, instruction graph, harness findings, latest runs, candidate reports, and decision trace references.
- Security and operations posture: secret scanning, redaction, artifact retention, backup/restore docs, security review docs, release checklist, release manifest, and CI templates.

## Release Principles

v1.0 should be boring in the right places:

- Preserve conservative defaults.
- Keep auto-apply narrow and auditable.
- Prefer robust local workflows over remote service ambitions.
- Make every generated change reviewable, reproducible, and reversible.
- Treat trace and model output as evidence, never authority.
- Keep documentation synchronized with the CLI and artifacts.
- Avoid scope creep into a general agent runtime, plugin marketplace, or team server.

## Milestone 1: 0.2.0 Stabilization And Documentation

Goal: make Tugboat pleasant and reliable for a new team to install, initialize, and run against a real trace.

Primary outcomes:

- A new user can go from `python -m pip install tugboat` to `tugboat optimize --repo . --trace traces/example.jsonl --suite all` with clear docs and actionable errors.
- Core commands produce consistent progress output and artifact references.
- Troubleshooting covers common setup, trace, `llmff`, eval, apply, rollback, MCP, daemon, and release issues.
- Real-world trace examples are documented and safe to share.

Scope:

- Documentation overhaul:
  - expand quickstart into individual adoption paths for solo users, teams, and CI;
  - complete best-practices guidance for instruction maps, skills, runbooks, and eval definitions;
  - add `docs/integrations.md` with integration notes for Codex, Claude Code, Cursor, Aider, Continue.dev, MCP clients, and CI systems;
  - publish example trace fixtures and explain what each fixture proves, starting with `tests/fixtures/traces/codex-local-session-export.jsonl`, `tests/fixtures/traces/claude-transcript.json`, `tests/fixtures/traces/mcp-session.jsonl`, and `tests/fixtures/traces/ci-failure.json`;
  - document policy defaults, auto-apply lanes, and safe opt-in procedures.
- CLI and UX polish:
  - improve `tugboat doctor` with actionable recommendations;
  - normalize error wording across audit/propose/eval/apply/rollback/daemon/MCP;
  - add clearer "next command" hints for blocked workflows;
  - improve `--help` summaries for commands with many safety flags.
- Testing and quality:
  - raise focused coverage on `apply`, `auto_apply`, `daemon`, `vcs`, and trace adapters;
  - add integration tests with realistic traces and malformed traces;
  - add fuzz or property-style coverage for unified diffs, bounded edit metadata, and policy gate classification.
- Trace ingestion:
  - harden auto-detection;
  - improve user-facing errors for missing files, malformed JSON, unsupported formats, and redaction failures;
  - preserve precise evidence IDs and active instruction snapshots.

Exit criteria:

- `docs/quickstart.md`, `docs/troubleshooting.md`, `docs/cli-reference.md`, `docs/apply-rollback.md`, `docs/auto-apply.md`, `docs/daemon-guide.md`, and integration docs pass docs contract tests.
- A credential-free install smoke test in an isolated virtual environment can install the current wheel, run `tugboat doctor --repo .`, `tugboat init --repo .`, `tugboat index --repo . --check`, `tugboat harness check --repo .`, and a fixture-backed proposal loop that writes `audit.json`, `candidate.json`, `eval-report.json`, `optimization-summary.json`, and `report.md`.
- Top setup and trace-ingestion failures return clean messages without Python tracebacks.
- Critical path coverage remains at or above 90% overall, with targeted tests for apply/rollback/auto-apply/daemon/vcs/trace-adapter failure modes.

## Milestone 2: 0.3.0 Review Intelligence

Goal: reduce review burden by making proposals more relevant, better grouped, and less repetitive.

Primary outcomes:

- Tugboat avoids repeating rejected proposals.
- Related drift and failure signals are grouped into higher-quality candidate patches.
- Review packets explain why a candidate matters, what evidence supports it, what risks remain, and how to reject it constructively.

Scope:

- Rejected-edit memory:
  - store rejected edit fingerprints by file, section, operator, category, and failure pattern;
  - suppress or down-rank repeated proposals;
  - surface suppression reasons in proposal and optimization artifacts.
- Drift clustering:
  - cluster related trace failures, stale docs, duplicate rules, and recurring harness issues;
  - propose consolidated patches when bounded edit budgets allow;
  - keep rejected clusters visible as optimizer memory.
- Proposal quality:
  - improve minimal diff generation and metadata validation;
  - expand safe edit operators only where policy and tests prove behavior;
  - enrich rationale, risk class, source references, and reviewer checklist fields.
- Review UX:
  - add or harden `inspect-decision` output;
  - add rejection reason templates that feed optimizer memory.

Exit criteria:

- Rejected proposals are recorded and later suppress materially similar candidates.
- Drift clustering produces stable, reviewable clusters with evidence IDs.
- Candidate artifacts explain rationale, risk, source evidence, expected behavior change, rollback plan, and review checklist.
- Tests cover rejected-memory suppression, cluster merging limits, and rejection reason propagation.

## Milestone 3: 0.4.0 Automation And Ecosystem

Goal: make local automation and ecosystem integration first-class without expanding authority beyond the v1 safety model.

Primary outcomes:

- The daemon is a mature local operator workflow, not a hidden mutation engine.
- MCP clients get richer read-only context.
- Teams have examples for common agent tools and CI patterns.
- Auto-apply has better monitoring, shadow mode, and pause controls.

Scope:

- Daemon maturation:
  - job prioritization and recovery;
  - queue visibility and stuck-job reporting;
  - local notification hooks that do not require a network service;
  - user-service guidance for Linux/macOS;
  - clearer daemon incident recovery.
- MCP expansion:
  - richer read-only tools for instruction graph queries, recent decisions, latest failed gates, active auto-apply lane status, and harness health;
  - preserve read-only default and repo allowlist behavior.
- Ecosystem:
  - compatibility guide for Codex, Claude Code, Cursor, Aider, Continue.dev, MCP clients, and CI systems;
  - starter repos and trace samples;
  - documented CI adoption patterns.
- Auto-apply monitoring:
  - canary or shadow mode for Class A candidates;
  - lane-level reports for `docs_hygiene` and `skill_improvement`;
  - pause controls by repo, lane, category, and incident state.

Exit criteria:

- Daemon recovery tests cover stale leases, interrupted jobs, read-only kill switch, bad trace input, and queue resume.
- MCP tools remain read-first and cannot directly mutate instruction files.
- Auto-apply reports show eligible, rejected, staged, applied, rolled back, and paused candidates by lane.
- Example repos and trace samples are linked from quickstart and integration docs.

## Milestone 4: 0.5.0 Through 0.8.0 Reliability And Hardening

Goal: earn production confidence across large repos, messy traces, concurrency, schema evolution, and security review.

Primary outcomes:

- Tugboat handles common production failure modes cleanly.
- Large repositories and large traces remain usable.
- Schema migrations are explicit and reversible.
- Security posture is documented and tested.

Scope:

- Failure mode analysis:
  - corrupted traces;
  - missing run artifacts;
  - interrupted `llmff` runs;
  - network failures in explicitly enabled provider-backed paths;
  - concurrent daemon and CLI operations;
  - dirty worktrees and stale base hashes;
  - rollback conflicts.
- Performance:
  - large instruction corpus indexing;
  - big trace files;
  - monorepo instruction scopes;
  - retention and cleanup of large `.sidecar` directories.
- Schema and compatibility:
  - migration tooling for `.sidecar/db.sqlite`;
  - artifact schema compatibility policy;
  - deprecation warnings before breaking CLI or artifact changes.
- Security:
  - formal threat model review;
  - adversarial trace fixtures;
  - provider-backed preflight checks;
  - redaction and secret scanning hardening;
  - read-only kill switch regression tests.
- Observability:
  - structured logging;
  - local metrics export options;
  - release evidence retention;
  - operator runbooks for incidents.

Exit criteria:

- Failure-injection tests cover the critical state transitions in audit/propose/eval/apply/rollback/daemon.
- Schema migration tests prove upgrade and blocked-downgrade behavior.
- Large fixture tests establish reasonable performance budgets.
- Security review records no open critical or high findings.

## Milestone 5: 0.9.0 Release Candidate

Goal: freeze the v1 surface and prove release readiness before the final 1.0 tag.

Primary outcomes:

- CLI and artifact contracts are stable enough to document as v1.
- Migration docs exist for 0.x users.
- Release process is reproducible and evidence-backed.
- Remaining release blockers are explicit and tracked.

Scope:

- CLI freeze:
  - document supported commands, options, exit codes, and artifact expectations;
  - define deprecation policy;
  - mark experimental surfaces clearly.
- Documentation freeze:
  - tutorial, how-to, reference, explanation, operations, security, and migration docs complete;
  - examples verified against the current release candidate.
- Release process:
  - build wheel;
  - run twine check;
  - install smoke test;
  - produce release manifest;
  - retain CI, coverage, security, and smoke evidence.
- Issue triage:
  - all P0 and P1 items closed or explicitly deferred;
  - known limitations documented.

Exit criteria:

- `python -m pytest --cov=src --cov-report=term-missing -q` passes at or above 90%.
- `tugboat ci --repo .`, `tugboat index --repo . --check`, and `tugboat harness check --repo .` pass.
- Wheel build, twine check, and installed CLI smoke pass.
- v1 migration guide and changelog are ready.

## Milestone 6: 1.0.0 Stable Release

Goal: publish a stable, production-grade release with clear adoption guidance and conservative defaults.

Release criteria:

- All v1 P0 features are complete and tested.
- CLI surface is frozen under the v1 compatibility policy.
- Documentation has been reviewed end to end.
- Test coverage remains at or above 90%, with stronger targeted coverage on critical paths.
- Release artifacts are signed or otherwise backed by retained release evidence.
- Migration guide from 0.x is published.
- Production announcement and case-study framing are updated.
- GitHub release includes changelog, install instructions, release notes, known limitations, and verification evidence.

Definition of done:

- New users can install Tugboat, initialize policy, index a repo, run the proposal loop, inspect artifacts, and understand how to apply or reject a candidate.
- Teams can adopt Tugboat in proposal-only CI without provider credentials.
- Operators can enable narrow Class A auto-apply with clear safety gates, monitoring, pause controls, and rollback.
- Maintainers can reproduce the release build and verify its evidence.

## Cross-Cutting Workstreams

Testing:

- Maintain 90%+ overall coverage.
- Add focused failure-injection tests for critical workflows.
- Add property/fuzz-style tests for diff parsing, bounded edit metadata, policy gates, and trace ingestion.

Security:

- Preserve secret scanning, policy gates, protected-domain classification, read-only kill switch, and provider opt-in.
- Complete formal threat model and release security review before v1.

Documentation:

- Keep tutorial, how-to, reference, explanation, and operations docs in sync.
- Treat examples as tested product surface, not marketing copy.

`llmff` integration:

- Maintain fixture-backed default behavior.
- Publish compatibility matrix for supported `llmff` versions and manifest contracts.
- Keep provider-backed execution explicitly policy-gated.

Community:

- Add contribution guide, issue templates, bug report template, feature request template, and security reporting instructions.
- Use GitHub milestones aligned to this roadmap.

Release process:

- Use `docs/ops/release-checklist.md`.
- Produce release manifest for every published release.
- Retain coverage, build, smoke, security, and install evidence.

## v1 Feature Tracks

Robust auto-update:

- Build from the current lane-based auto-apply model.
- Add preflight, shadow/canary mode, pause controls, monitoring, incident watch, and rollback watch.
- Keep broad auto-apply out of scope.

Trace adapter hardening:

- Expand real trace coverage.
- Add adversarial fixtures.
- Preserve active instruction snapshots and precise evidence IDs.

Provider-backed `llmff` manifests:

- Add reviewed manifests for audit, propose, eval, and acceptance summary.
- Require `llmff.allow_network: true`, reviewed provider policy, and pinned or reviewed manifest hashes.

Operator review UX:

- Continue improving operator review ergonomics on top of existing decision inspection, candidate comparison, risk explanation, rollback readiness, highest-impact summary, PR body, and rejection-template support.

Longitudinal metrics:

- Track acceptance rate, rejection rate, rollback rate, recurring incident rate, eval score trends, governance trends, mean changed lines, corpus growth, duplicate/conflicting rule count, stale-doc count, and queue health.

Token efficiency:

- Measure instruction corpus token footprint, active context pack token footprint, token delta per accepted edit, duplicate-token estimates, retrieval pack size, and token-growth policy violations.

Skill rewrite evaluation:

- Add skill-specific eval reports for `SKILL.md` rewrites.
- Check trigger preservation, executability, ambiguity, overfit risk, token footprint, required sections, forbidden sections, and safety weakening.

Team workflow:

- Harden PR mode, branch cleanup, ownership metadata, CI artifact links, and local fallback behavior.

Harness health:

- Improve duplicate/conflicting rule detection, stale references, large instruction warnings, runbook freshness, cleanup proposals, and cleanup eval suites.

Optional Zaxy bridge:

- Export accepted/rejected edit summaries and recurring failure patterns as advisory memory only.
- Never put Zaxy in the authority path for v1.

## Non-Goals For v1.0.0

- Public network daemon.
- Hosted web UI.
- General agent runtime.
- General memory system.
- Broad auto-apply beyond narrow Class A lanes.
- Plugin marketplace.
- Multi-user team server.
- Provider credential management through MCP.

## Risk Register

Scope creep:

- Mitigation: keep public daemon, hosted UI, broad auto-apply, plugin marketplace, and team server out of v1.

`llmff` dependency drift:

- Mitigation: fixture backend remains default, publish compatibility matrix, pin or review manifests, and test inspect/run contracts.

Breaking changes before v1:

- Mitigation: introduce deprecation policy during 0.9.0 and freeze documented CLI/artifact contracts before 1.0.

Low adoption:

- Mitigation: prioritize docs, examples, install smoke, quickstart clarity, and integration guides in 0.2.0.

Over-trusting automation:

- Mitigation: keep proposal-only default, require explicit lane policy, enforce VCS and rollback, preserve read-only kill switch, and document when not to enable auto-apply.

Security regressions:

- Mitigation: formal threat model, adversarial fixtures, redaction checks, provider opt-in checks, and release security review.
