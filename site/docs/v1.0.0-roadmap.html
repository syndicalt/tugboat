Tugboat v1.0.0 Roadmap

Purpose

This roadmap turns the 0.1.0 production foundation into a stable v1.0 release that teams can adopt as a local sidecar for governed agent instruction maintenance.

Tugboat v1.0 should be the trusted local system for continuously observing, evaluating, and maintaining the operational files that shape coding agents: CODEX.md, AGENTS.md, SKILL.md, runbooks, policies, eval definitions, and related harness config.

The core posture does not change for v1.0:

  • local-first operation;
  • proposal-only by default;
  • narrow Class A auto-apply only after explicit policy opt-in;
  • evidence-gated proposals;
  • human oversight as the primary control;
  • VCS-backed mutation and rollback;
  • llmff as the bounded pipeline runner, not Tugboat's authority layer.

Current Baseline

The 0.1.0 foundation already includes the major product skeleton:

  • CLI workflows for doctor, init, status, index, audit, propose, eval, optimize, apply, auto-apply, rollback, mcp, daemon, harness, ci, retention, and ops.
  • Trace ingestion and canonical episode capture for generic JSONL, Codex, Claude, MCP session, and CI failure traces.
  • Governed proposal loop with llmff inspect, file-backed llmff run artifacts, deterministic policy gates, eval reports, optimization summaries, and review reports.
  • VCS-gated apply modes: proposal, branch, commit, and pull request.
  • Rollback with recorded provenance, decision trace, apply plan, post-rollback hashes, and audit events.
  • Narrow auto-apply lanes for docs_hygiene and skill_improvement, disabled by default and gated by policy, confirmation, ledger metrics, eval evidence, VCS, and rollback.
  • Local daemon with queueing, run-once/cycle operations, read-only kill switch, stale lease handling, and local-only socket constraints.
  • Read-first MCP surface for status, instruction graph, harness findings, latest runs, candidate reports, and decision trace references.
  • Token delta metrics in eval and review surfaces, plus token footprint metrics in harness reporting.
  • SKILL.md rewrite checks through skill-specific eval report fields.
  • Security and operations posture: secret scanning, redaction, artifact retention, backup/restore docs, security review docs, release checklist, release manifest, and CI templates.

Remaining v1 work is maturity work, not a pivot. The roadmap should tighten the user journey, prove the safety model under failure, and make narrow automation observable enough that teams can trust it without turning Tugboat into a general runtime.

Release Principles

v1.0 should be boring in the right places:

  • Preserve conservative defaults.
  • Keep auto-apply narrow and auditable.
  • Prefer robust local workflows over remote service ambitions.
  • Make every generated change reviewable, reproducible, and reversible.
  • Treat trace and model output as evidence, never authority.
  • Keep documentation synchronized with the CLI and artifacts.
  • Avoid scope creep into a general agent runtime, plugin marketplace, or team server.
  • Treat lane burn-in thresholds as mechanical minimums, not a substitute for operator-reviewed production observation.

Burn-in policy:

  • Production teams should complete an operator-reviewed proposal-only observation period before enabling auto-apply.
  • The default recommended production observation period remains 30 days unless repo policy records a narrower risk decision with rollback ownership.
  • Auto-apply lane thresholds can be stricter than the default observation period; they should not silently weaken it.

Priority definitions:

  • P0 release blockers: work required before a stable 1.0.0 tag because it affects installation, safe proposal generation, governed mutation, rollback, documented contracts, or release evidence.

Milestone 1: 0.2.0 Stabilization And Documentation

Goal: make Tugboat pleasant and reliable for a new team to install, initialize, and run against a real trace.

Primary outcomes:

  • A new user can go from python -m pip install tugboat to tugboat optimize --repo . --trace traces/example.jsonl --suite all with clear docs and actionable errors.
  • Core commands produce consistent progress output and artifact references.
  • Troubleshooting covers common setup, trace, llmff, eval, apply, rollback, MCP, daemon, and release issues.
  • Real-world trace examples are documented and safe to share.

Scope:

  • Documentation overhaul:

    • - expand quickstart into individual adoption paths for solo users, teams, and CI; - complete best-practices guidance for instruction maps, skills, runbooks, and eval definitions; - add docs/integrations.md with integration notes for Codex, Claude Code, Cursor, Aider, Continue.dev, MCP clients, and CI systems; - publish example trace fixtures and explain what each fixture proves, starting with tests/fixtures/traces/codex-local-session-export.jsonl, tests/fixtures/traces/claude-transcript.json, tests/fixtures/traces/mcp-session.jsonl, and tests/fixtures/traces/ci-failure.json; - document policy defaults, auto-apply lanes, and safe opt-in procedures.

  • CLI and UX polish:

    • - improve tugboat doctor with actionable recommendations; - normalize error wording across audit/propose/eval/apply/rollback/daemon/MCP; - add clearer "next command" hints for blocked workflows; - improve --help summaries for commands with many safety flags.

  • Testing and quality:

    • - raise focused coverage on apply, auto_apply, daemon, vcs, and trace adapters; - add integration tests with realistic traces and malformed traces; - add fuzz or property-style coverage for unified diffs, bounded edit metadata, and policy gate classification.

  • Trace ingestion:

    • - harden auto-detection; - improve user-facing errors for missing files, malformed JSON, unsupported formats, and redaction failures; - preserve precise evidence IDs and active instruction snapshots.

Exit criteria:

  • docs/quickstart.md, docs/troubleshooting.md, docs/cli-reference.md, docs/apply-rollback.md, docs/auto-apply.md, docs/daemon-guide.md, and integration docs pass docs contract tests.
  • A credential-free install smoke test in an isolated virtual environment can install the current wheel, run tugboat doctor --repo ., tugboat init --repo ., tugboat index --repo . --check, tugboat harness check --repo ., and a fixture-backed proposal loop that writes audit.json, candidate.json, eval-report.json, optimization-summary.json, and report.md.
  • Top setup and trace-ingestion failures return clean messages without Python tracebacks.
  • Critical path coverage remains at or above 90% overall, with targeted tests for apply/rollback/auto-apply/daemon/vcs/trace-adapter failure modes.

P0 issue seeds:

  • New-user smoke: install, doctor, init, index --check, harness check, and fixture-backed optimize all complete without provider credentials.
  • Trace setup recovery: missing trace files, malformed JSON, unsupported formats, and redaction failures return actionable messages without Python tracebacks.
  • Docs contract: quickstart, troubleshooting, CLI reference, apply/rollback, auto-apply, daemon, integration, and best-practices docs are tested against current commands.

P1 issue seeds:

  • Add copyable examples for Codex, Claude Code, Cursor, Aider, Continue.dev, MCP clients, and CI.
  • Add a troubleshooting decision tree for common llmff, policy, trace, and VCS failures.

Milestone 2: 0.3.0 Review Intelligence

Goal: reduce review burden by making proposals more relevant, better grouped, and less repetitive.

Primary outcomes:

  • Tugboat avoids repeating rejected proposals.
  • Related drift and failure signals are grouped into higher-quality candidate patches.
  • Review packets explain why a candidate matters, what evidence supports it, what risks remain, and how to reject it constructively.

Scope:

  • Rejected-edit memory:

    • - store rejected edit fingerprints by file, section, operator, category, and failure pattern; - suppress or down-rank repeated proposals; - surface suppression reasons in proposal and optimization artifacts.

  • Drift clustering:

    • - cluster related trace failures, stale docs, duplicate rules, and recurring harness issues; - propose consolidated patches when bounded edit budgets allow; - keep rejected clusters visible as optimizer memory.

  • Proposal quality:

    • - improve minimal diff generation and metadata validation; - expand safe edit operators only where policy and tests prove behavior; - enrich rationale, risk class, source references, and reviewer checklist fields.

  • Review UX:

    • - add or harden inspect-decision output; - add rejection reason templates that feed optimizer memory.

Exit criteria:

  • Rejected proposals are recorded and later suppress materially similar candidates.
  • Drift clustering produces stable, reviewable clusters with evidence IDs.
  • Candidate artifacts explain rationale, risk, source evidence, expected behavior change, rollback plan, and review checklist.
  • Tests cover rejected-memory suppression, cluster merging limits, and rejection reason propagation.

P0 issue seeds:

  • Rejected-edit memory: record rejection fingerprints and suppress materially repeated candidates with visible reasons.
  • Drift clustering: consolidate recurring failures into bounded review clusters while preserving ordered evidence IDs.
  • Review evidence metadata: report candidate rationale, risk class, source evidence, expected behavior change, rollback plan, and reviewer checklist.

P1 issue seeds:

  • Review summary mode: highlight the highest-impact candidate changes and unresolved risks.
  • Rejection templates: make operator feedback structured enough to improve future optimizer behavior.

Milestone 3: 0.4.0 Automation And Ecosystem

Goal: make local automation and ecosystem integration first-class without expanding authority beyond the v1 safety model.

Primary outcomes:

  • The daemon is a mature local operator workflow, not a hidden mutation engine.
  • MCP clients get richer read-only context.
  • Teams have examples for common agent tools and CI patterns.
  • Auto-apply has better monitoring, shadow mode, and pause controls.

Scope:

  • Daemon maturation:

    • - job prioritization and recovery; - queue visibility and stuck-job reporting; - local notification hooks that do not require a network service; - user-service guidance for Linux/macOS; - clearer daemon incident recovery.

  • MCP expansion:

    • - richer read-only tools for instruction graph queries, recent decisions, latest failed gates, active auto-apply lane status, and harness health; - preserve read-only default and repo allowlist behavior.

  • Ecosystem:

    • - compatibility guide for Codex, Claude Code, Cursor, Aider, Continue.dev, MCP clients, and CI systems; - starter repos and trace samples; - documented CI adoption patterns.

  • Auto-apply monitoring:

    • - canary or shadow mode for Class A candidates; - lane-level reports for docs_hygiene and skill_improvement; - pause controls by repo, lane, category, and incident state.

  • Robust auto-update lifecycle:

    • - eligibility; - preflight; - shadow/canary; - VCS-backed staged update; - validation; - commit or pull request; - monitoring; - rollback.

Exit criteria:

  • Daemon recovery tests cover stale leases, interrupted jobs, read-only kill switch, bad trace input, and queue resume.
  • MCP tools remain read-first and cannot directly mutate instruction files.
  • Auto-apply reports show eligible, rejected, staged, applied, rolled back, and paused candidates by lane.
  • Auto-update tests prove stale base, dirty target, failed eval, failed governance, missing rollback metadata, active incident, paused lane/category/repo, and read-only kill switch all block update.
  • Successful auto-updates can be rolled back from recorded decision metadata.
  • Example repos and trace samples are linked from quickstart and integration docs.

P0 issue seeds:

  • Robust auto-update shadow/canary: evaluate Class A candidates in shadow mode before allowing policy-gated apply.
  • Robust auto-update pause controls: pause by repo, lane, category, incident state, and global read-only kill switch.
  • Robust auto-update rollback watch: monitor applied Class A changes for rollback triggers, failed gates, and abnormal rejection or rollback rates.
  • Daemon recovery: prove stale lease, interrupted job, bad trace, read-only kill switch, and queue resume behavior.
  • MCP read-only expansion: expose recent decisions, harness health, failed gates, lane status, and instruction graph queries without mutation tools.

P1 issue seeds:

  • Local notification hooks: support local-only operator notifications without adding a network service.
  • User service guides: document Linux and macOS service setup with clear stop and recovery commands.

Milestone 4: 0.5.0 Through 0.8.0 Reliability And Hardening

Goal: earn production confidence across large repos, messy traces, concurrency, schema evolution, and security review.

Primary outcomes:

  • Tugboat handles common production failure modes cleanly.
  • Large repositories and large traces remain usable.
  • Schema migrations are explicit and reversible.
  • Security posture is documented and tested.

Scope:

  • Failure mode analysis:

    • - corrupted traces; - missing run artifacts; - interrupted llmff runs; - network failures in explicitly enabled provider-backed paths; - concurrent daemon and CLI operations; - dirty worktrees and stale base hashes; - rollback conflicts.

  • Performance:

    • - large instruction corpus indexing; - big trace files; - monorepo instruction scopes; - retention and cleanup of large .sidecar directories.

  • Schema and compatibility:

    • - migration tooling for .sidecar/db.sqlite; - artifact schema compatibility policy; - deprecation warnings before breaking CLI or artifact changes.

  • Security:

    • - formal threat model review; - adversarial trace fixtures; - provider-backed preflight checks; - redaction and secret scanning hardening; - read-only kill switch regression tests.

  • Observability:

    • - structured logging; - local metrics export options; - release evidence retention; - operator runbooks for incidents.

Exit criteria:

  • Failure-injection tests cover the critical state transitions in audit/propose/eval/apply/rollback/daemon.
  • Schema migration tests prove upgrade and blocked-downgrade behavior.
  • Large fixture tests establish reasonable performance budgets.
  • Security review records no open critical or high findings.

P0 issue seeds:

  • Failure injection: cover interrupted audit/propose/eval/apply/rollback/daemon transitions and corrupted artifacts.
  • Schema migration: prove forward migration, rollback evidence, and blocked downgrade behavior for newer sidecars.
  • Security hardening: add adversarial trace fixtures, redaction regression tests, provider preflight tests, and read-only kill switch regressions.
  • Performance budgets: establish large-repo indexing, big-trace ingestion, and retention cleanup budgets that are deterministic enough for CI. Retention cleanup is bounded by retention.max_scan_files; instruction discovery is bounded by index.max_instruction_files before parsing or index DB writes; trace ingestion is bounded by trace.max_input_bytes before trace files or MCP episode payloads are persisted and by trace.max_events before unbounded trace bundles are built.

P1 issue seeds:

  • Monorepo instruction scopes: document and test multiple instruction roots without cross-scope mutation.
  • Structured observability: keep local metrics and JSONL events bounded, redacted, and useful for operators.

Milestone 5: 0.9.0 Release Candidate

Goal: freeze the v1 surface and prove release readiness before the final 1.0 tag.

Primary outcomes:

  • CLI and artifact contracts are stable enough to document as v1.
  • Migration docs exist for 0.x users.
  • Release process is reproducible and evidence-backed.
  • Remaining release blockers are explicit and tracked.

Scope:

  • CLI freeze:

    • - document supported commands, options, exit codes, and artifact expectations; - define deprecation policy; - mark experimental surfaces clearly.

  • Documentation freeze:

    • - tutorial, how-to, reference, explanation, operations, security, and migration docs complete; - examples verified against the current release candidate.

  • Release process:

    • - build wheel; - run twine check; - install smoke test; - produce release manifest; - retain CI, coverage, security, and smoke evidence.

  • Issue triage:

    • - all P0 and P1 items closed or explicitly deferred; - known limitations documented.

Exit criteria:

  • python -m pytest --cov=src --cov-report=term-missing -q passes at or above 90%.
  • tugboat ci --repo ., tugboat index --repo . --check, and tugboat harness check --repo . pass.
  • Wheel build, twine check, and installed CLI smoke pass.
  • Installed wheel in a clean virtualenv runs doctor, init, index, harness check, and fixture-backed optimize.
  • The smoke repo writes audit.json, candidate.json, eval-report.json, optimization-summary.json, and report.md.
  • ops release-manifest retains evidence for coverage, CI, harness, wheel build, twine check, installed smoke, security review, and commit identity.
  • v1 migration guide and changelog are ready.

P0 issue seeds:

  • Compatibility freeze: document supported commands, exit codes, artifact schemas, and deprecation policy.
  • Migration proof: verify upgrade from 0.x sidecars and blocked use of future-version sidecars.
  • Release evidence: retain CI, harness, coverage, build, twine, install-smoke, security, and release-manifest artifacts.

Milestone 6: 1.0.0 Stable Release

Goal: publish a stable, production-grade release with clear adoption guidance and conservative defaults.

Release criteria:

  • All v1 P0 features are complete and tested.
  • CLI surface is frozen under the v1 compatibility policy.
  • Documentation has been reviewed end to end.
  • Test coverage remains at or above 90%, with stronger targeted coverage on critical paths.
  • Release artifacts are signed or otherwise backed by retained release evidence.
  • Migration guide from 0.x is published.
  • Production announcement and case-study framing are updated.
  • GitHub release includes changelog, install instructions, release notes, known limitations, and verification evidence.

Definition of done:

  • New users can install Tugboat, initialize policy, index a repo, run the proposal loop, inspect artifacts, and understand how to apply or reject a candidate.
  • Teams can adopt Tugboat in proposal-only CI without provider credentials.
  • Operators can enable narrow Class A auto-apply with clear safety gates, monitoring, pause controls, and rollback.
  • Maintainers can reproduce the release build and verify its evidence.

P0 issue seeds:

  • Version and changelog: bump to 1.0.0, publish release notes, and include known limitations.
  • Signed or retained evidence: publish verification evidence sufficient to reproduce the release.
  • Production announcement: frame Tugboat as a local governed sidecar, not a broad self-modifying agent platform.

v1 Feature Tracks

Robust auto-update:

  • Build from the current lane-based auto-apply model.
  • Add preflight, shadow/canary mode, pause controls, monitoring, incident watch, and rollback watch.
  • Keep broad auto-apply out of scope.
  • The lifecycle is eligibility, preflight, shadow/canary, VCS-backed staged update, validation, commit or pull request, monitoring, and rollback.

Trace adapter hardening:

  • Expand real trace coverage.
  • Add adversarial fixtures.
  • Preserve active instruction snapshots and precise evidence IDs.

Provider-backed llmff manifests:

  • Add reviewed manifests for audit, propose, eval, and acceptance summary.
  • Require llmff.allow_network: true, reviewed provider policy, and pinned or reviewed manifest hashes.

Operator review UX:

  • Continue improving operator review ergonomics on top of existing decision inspection, candidate comparison, risk explanation, rollback readiness, highest-impact summary, PR body, and rejection-template support.

Longitudinal metrics:

  • Track acceptance rate, rejection rate, rollback rate, recurring incident rate, eval score trends, governance trends, mean changed lines, corpus growth, duplicate/conflicting rule count, stale-doc count, and queue health.

Token efficiency:

  • Measure instruction corpus token footprint, active context pack token footprint, token delta per accepted edit, duplicate-token estimates, retrieval pack size, and token-growth policy violations.
  • Gate token-growth regressions for instruction files where the candidate does not improve retrieval clarity, remove duplication, or increase measured eval quality.
  • Include token efficiency summaries in harness health and review reports without making token reduction override safety or correctness.
  • Require eval-report.json.metrics.instruction_token_delta or an equivalent explicit metric for auto-apply eligibility.
  • Block auto-apply when token growth exceeds policy thresholds.
  • Accept token reductions only when held-out behavior, safety checks, and operator legibility stay neutral or improve.

Skill rewrite evaluation:

  • Add skill-specific eval reports for SKILL.md rewrites.
  • Check trigger preservation, executability, ambiguity, overfit risk, token footprint, required sections, forbidden sections, and safety weakening.
  • Require rewrite candidates to preserve trigger preservation, executable steps, examples or fixtures when present, and explicit non-goals before they can be considered for Class A lanes.
  • Treat skill rewrite evaluation as proposal evidence first; auto-apply remains limited to narrow, policy-approved Class A improvements.
  • Failing trigger preservation, executability, ambiguity, overfit, token footprint, required sections, forbidden sections, or safety weakening forces passed: false and recommendation: reject.
  • skill_improvement auto-update requires a passing skill report, held-out skill behavior neutral-or-better, VCS rollback metadata, and lane thresholds.

Team workflow:

  • Harden PR mode, branch cleanup, ownership metadata, CI artifact links, and local fallback behavior.

Harness health:

  • Improve duplicate/conflicting rule detection, stale references, large instruction warnings, runbook freshness, cleanup proposals, and cleanup eval suites.

Optional Zaxy bridge:

  • Export accepted/rejected edit summaries and recurring failure patterns as advisory memory only.
  • Never put Zaxy in the authority path for v1.

Risk Register

Scope creep:

  • Mitigation: keep public daemon, hosted UI, broad auto-apply, plugin marketplace, and team server out of v1.

llmff dependency drift:

  • Mitigation: fixture backend remains default, publish compatibility matrix, pin or review manifests, and test inspect/run contracts.

Breaking changes before v1:

  • Mitigation: introduce deprecation policy during 0.9.0 and freeze documented CLI/artifact contracts before 1.0.

Low adoption:

  • Mitigation: prioritize docs, examples, install smoke, quickstart clarity, and integration guides in 0.2.0.

Over-trusting automation:

  • Mitigation: keep proposal-only default, require explicit lane policy, enforce VCS and rollback, preserve read-only kill switch, and document when not to enable auto-apply.

Security regressions:

  • Mitigation: formal threat model, adversarial fixtures, redaction checks, provider opt-in checks, and release security review.
